# ETF 动量轮动系统 — 核心逻辑开发文档

> 本文档面向开发者，说明系统核心逻辑、算法实现、数据流向及扩展方式。

---

## 一、项目概述

本系统基于 **Spring Boot 3.2 + Java 17 + MyBatis Plus + MySQL**，采用 **DDD 四层架构** 实现 ETF 动量轮动策略。

策略核心逻辑（来自策略手册）：

```
不买跌的（绝对动量 > 0）
  → 涨得稳的（上涨天数占比 > 50%）
    → 买最强的（相对动量排序）
      → 稳的多买（低波加权）
        → 趋势确认（均线过滤）
          → 大盘环境调整（中证 A500 120 日均线）
```

## 二、DDD 分层说明

### 2.1 分层结构

```
adapter           接口适配层：Controller、定时任务、外部数据爬虫入口
application       应用层：用例编排、DTO、Assembler、事件监听
domain            领域层：实体、值对象、领域服务、仓储接口（核心业务逻辑）
infrastructure    基础设施层：PO、Mapper、仓储实现、外部 API 客户端、配置
```

**依赖规则**：上层依赖下层；领域层不依赖任何其他层。

### 2.2 各层职责

| 层级 | 职责 | 不应包含 |
|------|------|----------|
| Adapter | 接收 HTTP 请求、渲染页面、触发定时任务 | 业务逻辑 |
| Application | 编排领域服务完成用例、事务边界、DTO 转换 | 算法细节 |
| Domain | 表达业务概念、实现算法、封装业务规则 | 数据库/HTTP 细节 |
| Infrastructure | 持久化、外部 API 调用、框架配置 | 业务规则 |

## 三、核心流程

### 3.1 数据采集流程

```
[定时任务 NavSyncJob] 或 [手动接口 /api/funds/sync]
        ↓
[FundDataSyncAppService]
        ↓
[NavDataFetcher 接口]
        ↓
├─ EastMoneyKlineNavFetcher（默认）：获取 ETF 历史 K 线
└─ SinaFinanceNavClient（备选）：获取最新实时行情
        ↓
[NavDataRepository.save / batchSave]
        ↓
fund_nav 表
```

**说明**：
- 默认使用东方财富 K 线接口（稳定、字段完整），新浪财经历史接口已不稳定，仅保留实时行情能力。
- 每只基金同步时按日期去重，避免重复插入。
- 同步完成后可调用 `validateNavCoverage(tradeDate)` 检查某日数据覆盖情况。

### 3.2 策略执行流程

```
[StrategyJob] 每周五 15:35 或 [/api/strategy/execute]
        ↓
[StrategyExecutionAppService.executeWeeklyStrategy(tradeDate)]
        ↓
1. 加载候选 ETF 池（排除中证 A500）
2. 批量加载历史净值
3. 判断大盘环境（中证 A500 收盘价 > 120 日均线？）
4. 对每个策略配置执行单策略：
     a. ETFScreeningService.screen()  四层筛选
     b. LowVolatilityWeightingService.allocate()  低波加权
     c. 根据大盘状态调整目标总仓位
     d. 生成 StrategyResult
5. PortfolioMergeService.merge() 合并两套策略结果
6. 保存并返回 RebalanceAdvice
```

**重要业务规则**（已确认）：
- 积极型、平衡型两套策略**独立计算**，同一 ETF 分别占有权重，不取高、不合并。
- 弱势市场：平衡型目标仓位减半，积极型清仓。

### 3.3 回测流程

```
[/api/backtest/run]
        ↓
[BacktestAppService.runBacktest(startDate, endDate)]
        ↓
1. 收集区间内的所有周五作为调仓日
2. 对每个调仓日：
     a. 执行策略得到 RebalanceAdvice
     b. 合并同一 ETF 的总暴露
     c. 计算持有到下一调仓日的组合收益
     d. 更新累计净值、最大回撤
3. 计算年化收益、夏普比率
4. 保存回测记录到 backtest_record
```

**收益计算方式**：
- 买入价 = 调仓日收盘价
- 卖出价 = 下一调仓日收盘价
- 单只收益 = 卖出价 / 买入价 - 1
- 组合收益 = Σ(单只收益 × 权重) / 总权重

**绩效指标**：
- 年化收益：`(期末净值)^(1/年数) - 1`
- 最大回撤：累计净值曲线峰值到谷值的最大跌幅
- 夏普比率：`(年化收益 - 无风险利率 2%) / 年化波动率`

## 四、关键算法实现

### 4.1 动量（Momentum）

```java
动量 = (当前收盘价 - N日前收盘价) / N日前收盘价 × 100%
```

- 短期动量窗口：积极型 5 日，平衡型 10 日
- 长期动量窗口：积极型 10 日，平衡型 20 日
- 绝对动量过滤使用长期动量 > 0
- 相对动量排序使用长期动量降序

### 4.2 上涨质量（Up Quality）

```java
上涨天数占比 = 窗口内上涨交易日数 / 总交易日数
```

- 上涨定义：当日收盘价 > 上一日收盘价
- 窗口与长期动量窗口一致
- 阈值 > 50%，过滤脉冲式上涨

### 4.3 波动率（Volatility）

```java
日收益率 = 当日收盘价 / 上一日收盘价 - 1
波动率 = 日收益率序列的标准差
```

- 波动率窗口：积极型 10 日，平衡型 20 日
- 用于低波加权：波动越小，权重越高

### 4.4 均线（Moving Average）

```java
均线 = 窗口内收盘价简单平均值
```

- 均线周期：积极型 20 日，平衡型 60 日
- 过滤条件：收盘价 > 均线

### 4.5 低波加权

```java
单只权重 = (1 / 波动率_i) / Σ(1 / 波动率)
```

步骤：
1. 计算每只 ETF 的倒数波动率
2. 归一化得到初始权重
3. 应用单只权重上限（平衡型 40%，积极型 50%）
4. 截断后二次归一化，确保权重总和 = 目标总仓位

### 4.6 大盘环境判断

- 基准：中证 A500 指数（000852）
- 判断：中证 A500 收盘价 > 120 日均线 → 强势，否则弱势
- 影响：
  - 强势：平衡型 70%，积极型 30%
  - 弱势：平衡型 35%（70% × 0.5），积极型 0%

### 4.7 止损

- 触发条件：单只 ETF 自进入建议持仓以来，历史最高收盘净值回撤 > 8%
- 操作：无条件清仓
- 当前 V1 仅生成建议，实际清仓由用户自行执行

## 六、扩展点

### 6.1 新增数据源

实现 `NavDataFetcher` 接口，并在实现类上加 `@Primary` 或修改注入方式：

```java
public interface NavDataFetcher {
    List<Nav> fetchHistory(String fundCode, LocalDate startDate, LocalDate endDate);
    Nav fetchLatest(String fundCode);
}
```

### 6.2 新增策略类型

1. 在 `StrategyType` 枚举中新增类型
2. 在 `strategy_config` 表中插入配置
3. 无需修改策略核心代码

### 6.3 新增风控规则

在 `RiskControlService` 中添加新的检查方法，并在 `StrategyExecutionAppService` 中调用。

### 6.4 真实持仓跟踪

当前系统只生成建议持仓。如需跟踪真实持仓：
1. 新增 `RealPosition` 聚合根和对应表
2. 新增 `TradeOrder` 记录实际交易
3. 在生成建议后对比实际持仓，输出调仓指令

---

## 七、注意事项

1. **数据精度**：所有金融计算使用 `BigDecimal`，避免浮点误差。
2. **日期处理**：使用 `java.time.LocalDate`，净值数据按交易日存储。
3. **数据源稳定性**：东方财富接口为本系统默认数据源，新浪接口作为备选。
4. **交易成本**：按确认规则，回测不含交易成本和滑点。
5. **调仓执行**：系统只生成建议，不直接对接券商交易接口。
6. **权重归一化**：低波加权应用单只上限后需二次归一化，确保总和准确。
