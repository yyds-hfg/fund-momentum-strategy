# 基金动量策略系统 — 项目核心逻辑与使用文档

> 本文档面向项目使用者与维护者，系统说明 ETF 动量轮动策略的核心逻辑、系统架构、功能接口与日常使用方法。

---

## 一、项目简介

本项目是一个基于 **Spring Boot 3.2 + Java 17 + MyBatis Plus + MySQL** 的 ETF 动量轮动策略系统，采用 **DDD 四层架构** 实现。

**策略定位**：通过“动量选股 + 上涨质量过滤 + 低波加权 + 均线确认 + 积极/平衡双策略混用”，在 A 股 ETF 市场中实现稳健的趋势跟踪轮动。

**当前版本边界**：
- 只生成调仓建议，不直接对接券商交易接口。
- 回测与绩效指标按收盘净值计算，不含交易成本与滑点。
- 数据依赖外部免费数据源（默认东方财富 K 线，新浪财经备选）。

## 二、核心策略逻辑

### 2.1 策略总览

策略核心流程可概括为：

```
不买跌的（绝对动量 > 0）
  → 涨得稳的（上涨天数占比 > 50%）
    → 买最强的（相对动量排序）
      → 稳的多买（低波加权）
        → 趋势确认（均线过滤）
          → 大盘环境调整（中证 A500 120 日均线）
```

### 2.2 五大核心因子

| 因子 | 作用 | 计算方式 |
|------|------|---------|
| **绝对动量** | 剔除弱势 ETF | 长期动量 > 0 才纳入候选 |
| **上涨质量** | 过滤“脉冲式上涨” | 窗口内上涨天数占比 > 50% |
| **相对动量** | 选出强势 ETF | 按长期动量降序取前 N 只 |
| **低波因子** | 决定仓位分配 | 权重 ∝ 1 / 波动率 |
| **均线因子** | 趋势确认 | 收盘价 > N 日均线 |

### 2.3 双策略混用

系统同时运行两套独立策略，最终合并输出调仓建议：

| 策略类型 | 资金占比 | 动量窗口 | 均线周期 | 最大持仓 | 单只上限 |
|---------|---------|---------|---------|---------|---------|
| **平衡型 (BALANCED)** | 70% | 10 / 20 日 | 60 日 | 3 只 | 40% |
| **积极型 (ACTIVE)** | 30% | 5 / 10 日 | 20 日 | 2 只 | 50% |

**重要规则**：两套策略**独立计算**，同一 ETF 在各自策略中分别占有权重，**不取高、不合并**。

### 2.4 大盘环境判断

以 **中证 A500（000852）** 为基准：

- **强势市场**：中证 A500 收盘价 > 120 日均线
  - 平衡型目标仓位：70%
  - 积极型目标仓位：30%
- **弱势市场**：中证 A500 收盘价 ≤ 120 日均线
  - 平衡型目标仓位：35%（70% × 0.5）
  - 积极型目标仓位：0%（清仓）

### 2.5 风控规则

| 规则 | 触发条件 | 操作 |
|------|---------|------|
| **止损** | 单只 ETF 自进入建议持仓以来，最高收盘净值回撤 > 8% | 无条件清仓 |
| **单只上限** | 权重超过策略配置上限 | 截断并二次归一化 |
| **冷静期** | 新买入未满 3 个交易日 | 不换仓（止损例外） |

## 三、系统架构

### 3.1 DDD 四层架构

```
┌─────────────────────────────────────────────────────────────┐
│  接口适配层 (Adapter)                                       │
│  REST Controller / 页面 / 定时任务 / 外部行情爬虫适配器       │
├─────────────────────────────────────────────────────────────┤
│  应用层 (Application)                                       │
│  AppService：编排领域服务，无业务逻辑，只负责用例流程         │
├─────────────────────────────────────────────────────────────┤
│  领域层 (Domain)   ★ 核心                                   │
│  聚合根、值对象、领域服务、仓储接口、领域事件                 │
├─────────────────────────────────────────────────────────────┤
│  基础设施层 (Infrastructure)                                │
│  仓储实现、MyBatis Mapper、数据库、外部 API 客户端、配置     │
└─────────────────────────────────────────────────────────────┘
```

**依赖原则**：上层依赖下层；领域层不依赖任何其他层，只通过仓储接口与领域事件与外部交互。

### 3.3 核心领域服务

| 服务 | 职责 |
|------|------|
| `ETFScreeningService` | 四层筛选：绝对动量、上涨质量、均线、相对动量 |
| `LowVolatilityWeightingService` | 低波加权、单只上限、二次归一化 |
| `MarketEnvironmentService` | 中证 A500 大盘环境判断 |
| `RiskControlService` | 止损判断、回撤计算、权重上限检查 |
| `PortfolioMergeService` | 合并两套策略结果为最终调仓建议 |

## 四、核心流程

### 4.1 数据同步流程

```
[定时任务 NavSyncJob] 或 [POST /api/funds/sync]
        ↓
[FundDataSyncAppService]
        ↓
[NavDataFetcher 接口]
        ↓
├─ EastMoneyKlineNavFetcher（默认）：获取 ETF 历史 K 线
└─ SinaFinanceNavClient（备选）：获取最新实时行情
        ↓
[NavDataRepository.save]
        ↓
fund_nav 表
```

**说明**：
- 每日收盘后（周一至周五 15:35）自动同步近 6 个月净值。
- 同步时按日期去重，避免重复插入。
- 首次使用前建议手动同步更长时间的历史数据（建议 ≥ 1 年）。

### 4.2 策略执行流程

```
[StrategyJob] 每周五 15:35 或 [POST /api/strategy/execute]
        ↓
[StrategyExecutionAppService.executeWeeklyStrategy(tradeDate)]
        ↓
1. 加载候选 ETF 池（排除中证 A500）
2. 批量加载历史净值
3. 判断大盘环境（中证 A500 收盘价 > 120 日均线）
4. 对每个启用策略独立执行：
     a. ETFScreeningService.screen()  四层筛选
     b. LowVolatilityWeightingService.allocate()  低波加权
     c. 根据大盘状态调整目标总仓位
     d. 生成 StrategyResult 并保存
5. PortfolioMergeService.merge() 合并两套策略结果
6. 返回 RebalanceAdvice（调仓建议）
```

### 4.3 回测流程

```
[POST /api/backtest/run]
        ↓
[BacktestAppService.runBacktest(startDate, endDate)]
        ↓
1. 收集区间内的所有周五作为调仓日
2. 对每个调仓日：
     a. 执行策略得到 RebalanceAdvice
     b. 合并同一 ETF 的总暴露
     c. 计算持有到下一调仓日的组合收益
     d. 更新累计净值、最大回撤
3. 计算年化收益、夏普比率
4. 保存回测记录到 backtest_record
```

## 八、核心算法公式

### 8.1 动量

```
动量 = (当前收盘价 - N日前收盘价) / N日前收盘价 × 100%
```

### 8.2 上涨质量

```
上涨天数占比 = 窗口内上涨交易日数 / 总交易日数
```

上涨定义：当日收盘价 > 上一日收盘价。

### 8.3 波动率

```
日收益率 = 当日收盘价 / 上一日收盘价 - 1
波动率 = 日收益率序列的标准差
```

### 8.4 均线

```
均线 = 窗口内收盘价简单平均值
```

### 8.5 低波加权

```
单只权重 = (1 / 波动率_i) / Σ(1 / 波动率)
```

应用单只上限后，截断并二次归一化，确保权重总和等于目标总仓位。

### 8.6 绩效指标

- **年化收益**：`(期末净值)^(1/年数) - 1`
- **最大回撤**：累计净值曲线峰值到谷值的最大跌幅
- **夏普比率**：`(年化收益 - 无风险利率 2%) / 年化波动率`

## 十、注意事项

1. **数据精度**：所有金融计算使用 `BigDecimal`，避免浮点误差。
2. **数据完整性**：策略计算前请确保候选 ETF 有足够历史净值，否则会被剔除。
3. **数据源稳定性**：东方财富接口为默认数据源，新浪接口作为备选。
4. **调仓执行**：系统只生成建议，实际交易需用户自行在券商平台执行。
5. **回测假设**：回测不含交易成本、滑点与分红再投资，按收盘净值直接计算。
6. **弱势市场**：积极型策略在弱势市场会清仓，平衡型仓位减半。
7. **权重合并**：同一 ETF 出现在两套策略中时，分别保留各自权重，展示为两条来源不同的持仓。

1. 在 `StrategyType` 枚举中新增类型。

在 `RiskControlService` 中添加新的检查方法，并在 `StrategyExecutionAppService` 中调用。

## 十二、常见问题

**Q1：为什么策略执行后没有持仓建议？**

可能原因：
- 净值数据不足，无法计算均线/波动率。
- 当前市场环境弱势，积极型策略清仓。
- 候选 ETF 均未通过动量/质量/均线过滤。

**Q2：如何修改策略参数？**

直接修改 `strategy_config` 表中的对应记录即可生效，无需重启应用。

**Q3：回测区间至少需要多长？**

至少需要包含两个周五，建议 1 年以上以获得有意义的绩效指标。

## 十三、一句话总结

> 用 70% 稳健仓位打底（平衡型）+ 30% 进攻仓位增强（积极型），通过“动量选股 + 上涨质量过滤 + 低波加权 + 均线过滤 + 大盘风控”五重机制，在控制风险的前提下争取跑赢市场。
